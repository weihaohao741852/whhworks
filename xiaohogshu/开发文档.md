作为一个 AI Agent 产品经理，开发一款集“爬取-总结-发布”于一体的自动化内容流工具，本质上是在构建一个**基于 LLM 的自主代理流水线（Agentic Workflow）**。

为了确保产品不仅能跑通，且具备商业竞争力和技术鲁棒性，你需要从以下四个维度进行准备：

---

## 1. 核心流程架构设计
你不仅是在做一个爬虫，而是在做一个**信息加工工厂**。你需要定义三个关键节点：

### 输入端（感知层）：多模态自动化爬取
* **信源管理**：支持 RSS、社交媒体（X/微博）、新闻网页、甚至是播客音频。
* **技术选型**：考虑使用网页解析工具（如 Firecrawl 或 Jina Reader），它们能将复杂的 HTML 直接转为大模型易读的 Markdown 格式。
* **绕过机制**：准备好代理 IP 池和验证码识别方案，应对反爬限制。

### 处理端（思考层）：长文本总结与重写
* **Prompt 策略**：设计“多级摘要”提示词。第一步提取事实，第二步根据目标风格重写。
* **模型路由**：简单的信息提取用较快的 $GPT-4o-mini$ 或 $Claude \ 3 \ Haiku$；深度评论或多源整合建议调用 $GPT-4o$ 或 $Claude \ 3.5 \ Sonnet$。

### 输出端（行动层）：多平台分发
* **API 适配**：对接微信公众号、知乎、小红书或 WordPress 的 API/模拟登录。
* **审核环节**：**必须加入人工介入点（Human-in-the-loop）**，在自动发布前由人进行合规性勾选和微调。

---

## 2. 技术储备与选型清单
作为 PM，你需要引导技术团队完成以下选型：

| 模块 | 推荐工具/方案 | 目的 |
| :--- | :--- | :--- |
| **工作流编排** | LangGraph, CrewAI, 或 Dify | 编排 Agent 的任务顺序、循环和条件判断。 |
| **结构化提取** | Instructor, Pydantic | 确保大模型输出的是固定的 JSON 格式，方便后端处理。 |
| **向量数据库** | Pinecone, Milvus | 用于存储历史爬取内容，避免重复抓取和查重。 |
| **内容审核** | 内容安全 API | 过滤政治敏感、色情及侵权内容。 |

---

## 3. 关键业务逻辑准备（PRD 核心）
在撰写需求文档时，请重点关注以下三个“坑”：

* **数据清洗（Clean Data）**：爬下来的数据含有很多广告、导航栏信息。你需要定义好清洗逻辑，否则会浪费大量的 Token 成本且干扰总结质量。
* **幻觉抑制（Hallucination Control）**：要求 Agent 在总结时必须附带“原文引用”，确保信息真实性。
* **版权与合规**：
    * **避风港原则**：自动发布的内容需注明来源。
    * **差异化处理**：通过 Prompt 要求 Agent 改变叙述视角（如从“客观报道”转为“犀利点评”），降低洗稿感。

---

## 4. 第一阶段：MVP（最小可行性产品）路径
不要试图一下做一个全能工具。建议先跑通一个垂直闭环：

1.  **确定垂直领域**：例如“AI 行业每日快讯”。
2.  **设定固定信源**：监控 5 个特定的英文科技媒体 RSS。
3.  **核心链路**：
    * 每 4 小时抓取一次原文。
    * 调用 LLM 翻译并提炼 200 字中文摘要。
    * 自动推送到你的 Slack 或 飞书 频道。
4.  **验证指标**：信息的准确率、总结是否有废话、发布的延迟时间。
